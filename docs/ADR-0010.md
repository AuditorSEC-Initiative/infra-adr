## Context

AuditorSEC operates as a Diia.City resident deeptech company providing
AI-augmented Web3 smart contract security (Audityzer scanner + LLM reasoning
+ NIST PQC hardening). As of Q1 2026:

- **73 DeFi exploits** in 2025 caused $15k–$50k average loss per protocol.
- LLM-Bridge integrates Telegram bots, FastAPI, and multi-agent pipelines
  that process **sensitive audit data** (contract code, vulnerability reports).
- NIS2 Directive (EU) 2022/2555 mandates risk-management (Art.21.2a),
  supply-chain security (Art.21.2d), and cryptographic controls (Art.21.2k).
- EU AI Act Art.14 requires **human oversight** mechanisms for high-risk AI
  systems: ability to monitor, intervene, and stop automated AI decisions.
- BRAVE1 Tier2 classification requires Ukrainian cybersecurity alignment
  (UHIP-2A phase).
- External AI chatbots (cloud-based) present data residency, supply-chain,
  and post-quantum risks if used without governance controls.

**Drivers for this ADR:**
1. Define baseline AI chatbot governance before specific vendor ADRs (ADR-0011+).
2. Establish human-in-loop requirements for all AI-assisted audit pipelines.
3. Mandate PQC (ML-KEM-768 / ML-DSA-65) for model communications at rest/transit.
4. Create enforcement hooks (governance.yaml, Woodpecker CI, ArgoCD).

---

## Decision

### 1. Human-in-Loop (EU AI Act Art.14)

All AI chatbot integrations in AuditorSEC MUST implement human oversight:

```yaml
human_in_loop:
  monitor: true       # Real-time Grafana/Prometheus dashboards
  intervene: true     # Manual override endpoints in FastAPI
  stop: true          # Kill-switch via ArgoCD rollback or NATS disconnect
  audit_log: true     # Immutable S3 logs (365-day retention)
  review_cycle: quarterly
```

- **Telegram bots** (@audityzerbot): All LLM-generated responses require
  confidence score >= 0.85; low-confidence outputs route to human reviewer.
- **FastAPI endpoints** (`/assess-risk`): Risk score >= HIGH triggers
  mandatory human review before report delivery.
- **Multi-agent orchestration**: Super Agents layer pauses on conflicting
  findings; human approval required to proceed.

### 2. Post-Quantum Cryptography (NIST FIPS 203/204)

All AI model communications MUST use PQC hardening:

```yaml
pqc:
  kem: ML-KEM-768      # FIPS 203 — key encapsulation
  dsa: ML-DSA-65       # FIPS 204 — digital signatures
  hybrid: X25519       # Classical fallback during transition
  encrypt_at_rest: true
  vault_backend: OpenBao
  profile: defi        # Options: defi | iot | defense
```

Ollama deployment (`deploy/ollama/values-pq.yaml`):
- Image tag: `ollama/ollama:latest-pq`
- Env: `OLLAMA_PQ_MODE=true`, `OLLAMA_ENCRYPT_AT_REST=true`
- Service: ClusterIP port 11434, namespace `ollama-system`
- Model: `mistral-nemo:pq` (primary), `qwen2.5-coder` (dev/test)
- Helm: `helm install ollama otwld/ollama -f values-pq.yaml -n ollama-system`

| Tool        | Status           | Scope                                    |

| Ollama      | ALLOWED_PREFERRED| All sensitive/classified workloads       |

### 4. Enforcement

```yaml
enforcement:

  ci_lint: Woodpecker CI             # PR-gate: blocks non-compliant configs
  gitops_sync: ArgoCD                # Continuous enforcement in K8s
  audit_logs: n8n + S3 immutable    # 365-day immutable trail
  monitoring: Prometheus + Grafana   # Real-time AI usage dashboards

```

## Consequences

### Positive
- Full NIS2 Art.21 + EU AI Act Art.14 compliance for AI pipelines.
- BRAVE1 Tier2 / USF EDGE / LLM-Bridge grant eligibility maintained.
- Data sovereignty via Ollama self-hosted PQC deployment (quantum-resistant).
- Blockchain-anchored audit trails via Diia.City compliance layer.
- Incident register with CERT-UA reporting and heat-maps enabled.

### Negative / Trade-offs
- Ollama GPU requirement (NVIDIA) increases infra cost vs cloud APIs.
- Human review latency (~2–5 min) for HIGH-risk findings vs fully automated.
- PQC overhead: ~15% CPU increase for ML-KEM/ML-DSA operations.

### Risks
- `qwen2.5-coder` model not yet PQC-hardened for prod; use only in dev.
- n8n automation audit logs require S3 bucket policy enforcement.

## Quick Start: Ollama PQ Local Test

```bash
# 1. Clone llm-bridge
git clone https://github.com/AuditorSEC-Initiative/auditorsec-llm-bridge
cd auditorsec-llm-bridge

# 2. Start Ollama with PQ mode
docker run -d -p 11434:11434 \
  -e OLLAMA_PQ_MODE=true \
  -e OLLAMA_ENCRYPT_AT_REST=true \
  --name ollama-pq \
  ollama/ollama:latest-pq

# 3. Pull and run qwen2.5-coder (dev)
ollama run qwen2.5-coder

# 4. Test neural CTI integration
curl -X POST http://localhost:11434/api/generate \
  -H 'Content-Type: application/json' \
  -d '{"model": "qwen2.5-coder", "prompt": "Audit this Solidity function for reentrancy."}'

# 5. Production Helm deploy (k3s)
helm install ollama otwld/ollama \
  -f infra-adr/deploy/ollama/values-pq.yaml \
  -n ollama-system
```
